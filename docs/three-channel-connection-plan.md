# RobotDog Studio 三通道连接与固件下载计划

## 1. 方案结论

RobotDog Studio 采用三条职责明确、逐级兜底的连接通道：

| 通道 | 面向用户 | 主要职责 | 正常使用频率 |
|---|---|---|---|
| 高质量蓝牙串口 | 学生 | 无线控制、参数调试、状态与 CCD 遥测、进入升级模式 | 日常使用 |
| 板载有线串口 | 学生 | IAP 固件下载、校验、重启和升级失败重试 | 修改程序后 |
| WCH-Link | 教师/维护人员 | 首次写入 Bootloader、底层诊断和损坏救砖 | 极少使用 |

这套方案的产品目标是：机器马可以留在地面运行，电脑放在桌面无线观察和调参；需要更新程序时只接一根 USB 线；学生无需接触 WCH-Link、OpenOCD、地址和命令行参数。

WCH-Link 不能完全取消。它是 Bootloader 被覆盖、时钟或选项字节错误、芯片无法启动等情况下的最终恢复路径，但默认隐藏在教师模式中。

## 2. 必须先确认的硬件条件

### 2.1 推荐连接方式

- 蓝牙模块独占一个 MCU UART，作为运行态链路。
- 板载 USB 转串口芯片独占另一个 MCU UART，作为 Bootloader/IAP 链路。
- WCH-Link 保留 SWD 调试与下载接口。
- USB 转串口芯片应具有稳定的 VID/PID；量产时优先选带唯一序列号的器件。
- 板上保留一个不易误触但可操作的“升级/恢复”按键，允许在应用固件损坏时强制进入 IAP。

现有固件可暂时把 USART3 PB10/PB11 作为蓝牙运行态链路。IAP 使用哪个 UART，需要结合当前 PCB 原理图和板载 USB 转串口连接确定；不能仅因为官方示例使用 USART3 就直接照搬引脚。

### 2.2 不推荐的连接方式

蓝牙模块和 USB 转串口若直接并联到同一组 MCU TX/RX，可能出现发送端电平争用、蓝牙启动信息污染 IAP、两个上位机端口同时打开等问题。若 PCB 已经这样设计，需要增加硬件隔离，至少满足以下一种条件：

- 模拟开关/多路复用器在运行态和升级态之间切换；
- MCU 控制蓝牙模块 EN，使 IAP 期间蓝牙 TX 进入高阻；
- 跳帽或可靠的串联隔离设计，且升级流程能检测当前模式。

在原理图确认前，上位机开发可以完成模拟器、状态机和协议，但不能把真实串口下载标记为硬件验收通过。

## 3. 用户体验

### 3.1 学生日常调试

1. 首次使用时在 Windows 中配对蓝牙模块。
2. RobotDog Studio 通过协议握手识别机器马，而不是只凭 COM 编号猜测设备。
3. 学生点击“无线连接”，之后可以控制动作、修改参数、观察 CCD 曲线和日志。
4. 软件记忆设备身份，下次启动自动尝试重连，但重连后绝不自动恢复运动。
5. 蓝牙断开或心跳超时后，下位机动作看门狗立即停止运动。

### 3.2 学生下载固件

界面只显示一个主操作：“下载到小马”。完整流程由 Main 进程执行：

```text
检查固件包
  → 提示连接 USB 下载线
  → 通过蓝牙发送停止并确认静止
  → 请求应用进入 IAP，或提示按升级键
  → 识别有线下载端口并完成 Bootloader 握手
  → 擦除 APP 区
  → 分块写入并逐块确认
  → 整体校验
  → 标记 APP 有效并复位
  → 验证新固件版本
  → 恢复蓝牙连接
```

若机器马未连接蓝牙，也允许通过“USB + 升级键”进入下载。烧录失败后 Bootloader 必须保持可连接状态，界面提供“重新下载”，不要求学生立即使用 WCH-Link。

### 3.3 教师救砖

教师模式提供“WCH-Link 恢复”向导：检测探针、确认目标芯片、写入完整受信镜像、校验、复位并验证。该入口默认隐藏，不能由学生模式或 AI 自动调用。

## 4. 上位机架构

### 4.1 领域模型

上位机不再使用一个模糊的“串口服务”承担全部职责，而是分为：

- `DeviceRegistry`：记录机器马设备 ID、昵称、板型、蓝牙端口身份、USB 下载口身份和最近固件版本。
- `RuntimeConnectionManager`：管理蓝牙扫描、握手、心跳、重连、命令队列和遥测限流。
- `BluetoothSerialTransport`：只负责 Windows 蓝牙 SPP 串口的收发与端口生命周期。
- `FirmwarePackageService`：验证固件包、板型、Flash 区域、版本、长度和哈希。
- `FirmwareUpdateOrchestrator`：编排停止、进入 IAP、端口切换、下载、校验、重启和重连。
- `WiredIapTransport`：实现有线串口 Bootloader 的握手、分块传输、重试和进度。
- `WchLinkRecoveryAdapter`：封装固定参数的 WCH OpenOCD 恢复流程。
- `PortWatcher`：监听 COM 端口出现、消失和编号变化。

Renderer 只收到设备状态、步骤进度和可执行的语义动作，不接触串口路径、固件绝对路径、IAP 原始帧或 OpenOCD 参数。

### 4.2 连接状态

无线运行态状态机：

```text
disconnected → discovering → connecting → handshaking → ready
      ↑                                                   ↓
      └──────── retrying ← degraded ← heartbeat_lost ─────┘
```

固件更新状态机：

```text
idle → preflight → stopping → waiting_for_usb → entering_iap
     → bootloader_handshake → erasing → writing → verifying
     → rebooting → validating_app → completed
```

`erasing` 到 `verifying` 属于关键区间。界面可以请求取消，但上位机只能在协议定义的安全边界停止，不能粗暴关闭进程或串口。

### 4.3 IPC 规划

新增或调整以下窄接口：

- `device.list/pairHint/forget`
- `runtime.connect/disconnect/command`
- `runtime.telemetry.start/stop`
- `firmware.build/inspectPackage`
- `firmware.update.start/cancel/retry`
- `firmware.recovery.preflight/start`
- `settings.bluetooth/updatePort/recoveryTool`

事件统一为：

- `device.connectionChanged`
- `runtime.telemetry/log`
- `firmware.updateProgress`
- `firmware.recoveryProgress`
- `operation.error`

所有 IPC 参数使用 Zod 校验，Renderer 只能传内部 `deviceId`、`artifactId` 和枚举动作。

## 5. 协议规划

### 5.1 蓝牙运行态协议

运行态协议承载握手、能力查询、控制、状态、参数和 CCD，不承载 Flash 写入。首版可继续兼容 `@RDS1` 行协议，但需要增加：

- 稳定的设备 ID、板型、协议版本和固件版本；
- 请求序号、应答关联与结构化错误码；
- 心跳和动作租约；
- 遥测订阅、速率协商与拥塞降级；
- 进入 IAP 的受控命令和确认应答。

115200 波特率下，128 点 CCD 使用冗长 ASCII 可能接近链路上限。首版将 CCD 限制在稳定帧率并监控队列延迟；后续协议版本使用带长度和 CRC 的紧凑二进制遥测帧，控制命令仍保持可诊断性。

持续运动期间由上位机发送心跳，固件建议在 300–500ms 未收到有效心跳时停止。现有 3 秒上位机动作租约可作为第二层限制，但不能代替固件短时看门狗。

### 5.2 有线 IAP 协议

IAP Bootloader 位于受保护 Flash 区，学生固件包只能更新 APP 区。固件镜像头至少包含：

- 魔数和格式版本；
- 目标板型、芯片和硬件版本；
- APP 起始地址、镜像长度和固件版本；
- 镜像 CRC32 和 SHA-256；
- 构建 ID 与协议兼容范围。

传输使用同步头、命令、序号、长度、负载和 CRC。支持能力查询、擦除 APP、分块写入、读取校验状态、完成、复位和错误码。每个写入块都要可重试，整包校验通过后才写入“APP 有效”标记。上位机校验包的 SHA-256，Bootloader 至少完成适合 MCU 的 CRC32 全镜像校验。

Bootloader 必须满足：

- 上电检查升级按键、升级标记和 APP 有效性；
- APP 无效时不跳转，并持续等待重新下载；
- APP 有效且没有升级请求时快速跳转，避免 Bootloader 给正常开机增加明显等待；
- 不允许 IAP 命令擦除或覆盖 Bootloader；
- 跳转前恢复中断、时钟和外设到约定状态；
- 下载期间关闭舵机输出或进入明确安全姿态；
- 输出稳定设备 ID、Bootloader 版本和板型。

CH32V20x 官方 USB/UART IAP 示例可作为移植起点，但最终 APP 偏移和引脚必须由实际链接脚本、Bootloader 体积与 PCB 共同确定。

## 6. 端口识别与配对策略

- 不把 `COM5` 一类编号作为设备身份，因为重启和换 USB 口后编号会变化。
- 有线口优先使用 USB VID/PID、USB 序列号和 Bootloader 握手三重识别。
- 蓝牙 SPP 在 Windows 上可能生成传入/传出两个 COM 口，上位机应逐个低频握手，并记住真正响应 RDS 协议的端口。
- 设备握手返回相同的稳定 `deviceId`，使蓝牙口和 USB 下载口可以归属于同一台机器马。
- 同时存在多台设备时，下载前必须校验目标设备 ID；无法证明一致时要求用户在列表中确认，不自动猜测。

## 7. 界面计划

顶部设备区显示两个独立状态：

- “无线调试”：未配对、正在连接、已连接、信号/延迟异常；
- “USB 下载”：未连接、已识别、Bootloader 模式、正在下载。

学生模式：

- 主按钮为“无线连接”“下载到小马”“停止”；
- 下载向导使用清晰步骤图，不出现地址、波特率和 OpenOCD；
- 蓝牙正常但 USB 未连接时，下载按钮可点击，点击后进入接线引导；
- 下载结束后显示“已安装版本”和一次运行态握手结果。

教师模式：

- 显示原始端口、协议版本、设备 ID、丢包率、重试次数和诊断导出；
- 提供“强制进入升级模式”和“WCH-Link 恢复”；
- 恢复操作需要明确确认，并使用只读板卡配置。

## 8. 分阶段实施

### 阶段 A：双链路模拟与契约

- 将现有模拟串口拆分为蓝牙运行态模拟器和有线 IAP 模拟器。
- 完成设备注册、两套状态机、端口变化模拟和 IPC 契约。
- 完成下载向导 UI，不连接真实硬件。

验收：可模拟蓝牙断线、USB 插拔、端口号变化、写入失败、重试和成功重连；Renderer 无本机串口与进程权限。

### 阶段 B：真实蓝牙调试

- 接入 `serialport` 和 Windows 蓝牙 SPP 枚举。
- 实现握手、设备绑定、心跳、动作安全、状态和 CCD 限流。
- 保留旧固件适配器，新增正式 RDS 协议适配器。

验收：机器马在地面、电脑在桌面连续调试 30 分钟；断线后固件在目标看门狗时间内停止；恢复连接后不自动继续动作；CCD 与命令不会互相长期阻塞。

### 阶段 C：有线串口 IAP

- 下位机移植并保护 Bootloader，调整 APP 链接地址和固件包格式。
- 上位机实现 IAP 传输、端口检测、包校验、重试和版本验证。
- 打通“蓝牙停止并进入 IAP → USB 下载 → 重启 → 蓝牙重连”。

验收：连续下载 100 次无不可恢复失败；在擦除、写入和校验阶段分别模拟拔线/断电后，重新接线仍可下载；错误板型镜像 100% 被拒绝。

### 阶段 D：WCH-Link 教师恢复

- 使用内置 WCH OpenOCD 和只读板卡配置写入完整镜像。
- 加入驱动检测、探针检查、恢复日志和恢复后验证。

验收：空片、APP 损坏和 Bootloader 损坏三种场景均能由教师向导恢复；学生模式无法调用恢复接口。

### 阶段 E：发布与课堂压力测试

- 在 Windows 11 标准用户、中文用户名、离线环境和多台机器马场景测试。
- 测试蓝牙配对丢失、多个 SPP 端口、多个 USB 下载口、电脑休眠和应用升级。
- 固定工具与固件包版本，记录来源、许可证和 SHA-256。

验收：教师能在不接触命令行的情况下为整组设备完成配对、下载和个别救砖；学生的默认流程不出现开发工具术语。

## 9. 测试矩阵

- 蓝牙：分片、粘包、噪声、延迟、丢包、断线、端口被占用、重复配对和双 COM 口。
- 遥测：CCD 峰值流量、命令抢占、队列上限、降帧和过期帧丢弃。
- 安全：运动中断线、应用崩溃、电脑休眠、退出软件和设备复位。
- IAP：错包、错板型、CRC 错误、重复块、乱序、超时、拔线、断电和重试。
- 识别：COM 编号变化、多台同型号设备、无 USB 序列号和蓝牙重命名。
- WCH-Link：未连接、驱动缺失、目标不匹配、写入失败和校验失败。

## 10. 开发前需要从硬件与固件侧取得的信息

1. PCB 原理图中蓝牙模块、板载 USB 转串口和 MCU UART 的实际连接与电源/使能方式。
2. 蓝牙模块型号、固件版本、Windows SPP 行为、可配置波特率和唯一身份能力。
3. USB 转串口芯片型号、VID/PID、序列号能力及自动复位/升级控制线。
4. CH32V203 的具体容量型号、当前链接脚本、Flash 分区和写保护策略。
5. 强制进入升级模式可使用的按键或 GPIO。
6. 正式固件的命令行构建入口和 BIN 输出约定。

这些信息不阻塞阶段 A，但它们是阶段 B/C 真机实现和验收的入口条件。
