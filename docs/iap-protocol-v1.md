# RobotDog IAP 二进制协议 v1

本文定义 RobotDog Studio 与板载有线 Bootloader 之间的逻辑协议。UART 外设、引脚、USB 转串口芯片、波特率和自动复位线路均属于板卡适配配置，当前不写入协议常量。

## 帧格式

所有多字节整数使用小端序：

| 偏移 | 长度 | 字段 | 说明 |
|---|---:|---|---|
| 0 | 4 | Sync | ASCII `RDSI` |
| 4 | 1 | Version | 当前为 `1` |
| 5 | 1 | Command | 命令或响应类型 |
| 6 | 2 | Sequence | 请求序号，范围 0–65535 |
| 8 | 2 | Length | Payload 字节数，最大 1024 |
| 10 | N | Payload | 命令数据 |
| 10+N | 4 | CRC32 | 从 Version 到 Payload 的标准 CRC32 |

CRC32 使用多项式 `0xEDB88320`、初值 `0xFFFFFFFF`、最终异或 `0xFFFFFFFF`。标准检查字符串 `123456789` 的结果为 `0xCBF43926`。

## 命令编号

| 命令 | 编号 | 作用 |
|---|---:|---|
| HELLO | `0x01` | 查询设备 ID、板型、Bootloader 版本和能力 |
| ERASE_APP | `0x10` | 擦除 APP 允许区域 |
| WRITE_BLOCK | `0x11` | 按地址和序号写入固件块 |
| VERIFY | `0x12` | 校验完整 APP 长度与 CRC32 |
| COMPLETE | `0x13` | 校验通过后写入 APP 有效标记 |
| REBOOT | `0x14` | 复位并尝试启动 APP |
| RESPONSE | `0x80` | 成功响应 |
| ERROR | `0x81` | 带结构化错误码的失败响应 |

同一请求的响应必须复用请求序号。上位机按序号等待应答，超时可以重发同一写入块；Bootloader 应把相同序号和地址的重复写入视为幂等操作。

## 流解析要求

- 必须处理任意分片、多个帧粘连和同步头前噪声。
- CRC 错误的帧直接丢弃，不能执行其中命令。
- Length 超过 1024 时丢弃当前候选同步头并继续搜索。
- 不因一个损坏帧丢弃其后的完整合法帧。
- 写入、擦除和完成命令必须再次检查目标地址，不能只信任上位机固件包。

参考实现位于 `src/shared/iap-protocol.ts`，同时供 Main 模拟器、协议测试和未来真实串口适配器使用。

## 固件包校验

上位机在发送 ERASE_APP 前检查：格式版本、板型、芯片、硬件版本、APP 起始地址、镜像长度、CRC32 和 SHA-256。Bootloader 至少独立检查允许地址范围、镜像长度和完整 CRC32；APP 整体校验成功前不得写入有效标记。

实际 Flash 区间由只读板卡配置提供。当前代码不把尚未确认的 CH32V203 容量、Bootloader 大小或 UART 引脚写死。
