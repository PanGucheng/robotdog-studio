# RobotDog Studio 下一阶段详细计划

更新日期：2026-06-20

## 1. 当前基线

目前已经完成：

- Electron Main / Preload / Renderer 安全隔离；
- 模拟机器马、CCD、动作控制和软件急停；
- 蓝牙运行态、有线 IAP、WCH-Link 教师恢复的模拟状态机；
- 内置沁恒 GCC12 / OpenOCD 检测与外部 CH32V203 工程命令行构建；
- 固定 Reasonix v1.9.1 ACP、连续对话、Markdown、权限审批和取消；
- AI 候选 worktree、Diff 白名单、预检编译、哈希绑定、原子应用、Git 历史与撤销；
- 应用失败、重启中断、越权修改、候选篡改等自动化恢复测试。

当前最大的闭环缺口是：学生修改虽然可以安全应用并通过单元预检，但尚未被可靠地映射到一份固定版本的完整 CH32V203 固件工程，也还没有可交付给普通教师电脑的 Windows 安装包。

新增评审还发现五项必须在发布前解决的产品问题：Reasonix 缺少面向学生和本工程的完整内置提示词；缺少受控代码编辑器；工具修改审批过于频繁；高 DPI 屏幕字号和布局偏小；功能术语晦涩且没有操作示范。

因此下一阶段先补齐教学交互和显示基础，再完成“完整固件产物 + Windows 可交付版本”，暂不接入真实蓝牙串口、IAP 或 WCH-Link。

## 2. 阶段目标

形成以下纯软件闭环：

```text
适合当前屏幕和学生能力的工作台
  → AI 对话修改或手动编辑学生代码
  → 候选区内受控工具自动执行
  → 整轮修改统一审批一次
  → 预检与编译错误中文解释
  → 应用到学生 Git 工作区
  → 合入固定固件基线的隔离构建目录
  → 生成 ELF / HEX / BIN / MAP 和构建证明
  → 在界面中查看、导出或进入模拟下载
  → 安装包在无 Node.js、无 MounRiver 的 Windows 11 电脑运行
```

阶段完成后，“编译成功”必须表示完整固件已完成编译和链接，不再只是学生代码单元预检。

## 3. 明确不做的内容

本阶段继续跳过：

- 真实蓝牙串口枚举、配对、收发和重连；
- 真实 IAP 擦除、写入、校验与 Bootloader 联调；
- 真实 WCH-Link/OpenOCD 烧录；
- 引脚、电气隔离、串口复用和通信波特率的最终决定；
- 真机动作安全验收。

上述功能保持现有模拟实现，直到原理图、Bootloader 和目标板可用于验收。

## 4. 实施顺序

### 阶段 G1：教学型 AI、统一审批与显示适配基础

状态（2026-06-20）：学生默认路径已完成。内置提示词采用 `robotdog-student-v1.0.0` 并记录 SHA-256，包含年龄层、工程地图、WCH GCC 预检、编译错误解释、最小修改和安全边界；当前策略版本作为动态上下文注入。白名单候选编辑自动允许，越权工具自动拒绝，只有真实需求选择才询问，整轮结束统一展示 Diff。界面提供 100%/125%/150%/175% 缩放、2K/4K 自动推荐、持久化设置和缩放后容器重排。教师逐工具诊断开关留到 G5 设置中心一并实现。

目标：先解决 AI 不懂项目背景、审批打断频繁、2K/4K 屏幕字体过小的问题，为后续代码编辑器和完整构建提供稳定交互底座。

#### G1.1 版本化内置提示词

新增由 Studio 管理、可测试和可升级的系统提示词。它至少包含：

1. **身份与对象**：AI 是机器马巡线课程助教，主要面对高年级小学生；默认假设学生只理解变量、条件和简单函数。
2. **解释方式**：先用生活化中文说明目标，再说明改了什么、为什么、可能看到什么现象；首次出现专业词时给一句解释，不堆砌术语。
3. **工程地图**：明确学生可读、可改、只读和禁止访问的目录，以及 `student_control.c`、头文件、巡线 YAML 各自用途。
4. **学生接口**：列出允许调用的函数、参数单位、有效范围和典型例子；明确禁止修改硬件引脚、时钟、通信协议、Bootloader 和构建脚本。
5. **工具链工作流**：先读相关文件，形成简短修改计划，在隔离候选区修改，运行受控预检，读取结构化诊断，再向学生解释结果。
6. **修改原则**：一次只完成用户提出的目标；优先最小改动；不顺手重构；不伪造编译、下载或真机测试成功。
7. **编译错误解释**：指出相对文件、行号、错误含义、最可能原因和建议修法；先解释学生代码，再区分工具链或工程错误。
8. **输出格式**：完成时固定给出修改文件、修改前后、原因、预检结果和下一步，语言适合课堂阅读。
9. **安全边界**：提示词只负责指导模型，Main 的白名单、路径和 Diff 校验仍是最终安全边界，不能用提示词代替强制策略。

提示词拆成“固定教学规则 + 固件接口说明 + 当前工作区动态上下文”，记录版本号和哈希。固件接口或课程难度变化时不需要重写整个提示词；旧对话记录当时使用的提示词版本，便于复现。

#### G1.2 一轮对话只统一确认一次

将当前逐工具确认改为分层权限：

- 读取、搜索候选工作区自动允许；
- 对白名单内学生文件的创建或修改自动允许，但只发生在隔离候选区；
- 删除、重命名、二进制文件、越界路径、Shell、网络、烧录、串口和密钥访问直接拒绝，不弹出诱导学生放行的对话框；
- AI 完成整轮修改并通过策略校验后，界面汇总全部文件和 Diff，只让用户选择一次“应用这次修改”或“放弃”；
- 教师诊断模式可选保留逐步审批，用于调试 Reasonix，不作为学生默认流程。

统一确认不包含真正的烧录操作；未来下载固件仍需独立、明确的安全确认。

#### G1.3 高 DPI 与可读性基础

1. 建立统一字号、间距和控件尺寸 token，正文默认不低于 14px，主要操作和学生解释优先使用 16px；移除当前大量 9–11px 固定字号。
2. 使用 `rem`、`clamp()` 和容器断点重构布局，不用简单放大整个页面掩盖拥挤问题。
3. 增加界面缩放设置：100%、125%、150%、175%，跟随系统缩放作为默认值，并持久化每台电脑的选择。
4. 在空间不足时让对话、工作台、图表和按钮真实重排；不允许关键按钮被截断或只能通过横向滚动访问。
5. 保持键盘焦点可见、点击目标不小于 40px，并尊重系统“减少动画”设置。
6. 视觉语言继续采用“机器马比赛维修站”，但把最醒目的结构用于一条可理解的任务轨道：`提出想法 → 看懂修改 → 生成程序 → 连接小马`，其余界面保持安静。

验收：

- 固定提示词契约测试覆盖学生语气、工程边界、工具链步骤和编译错误解释；
- 一次包含多个允许文件修改的对话，过程中没有重复确认，结束时只出现一次总 Diff 审批；
- 越权工具调用自动拒绝且正式工作区不变；
- 在 1920×1080、2560×1440、3840×2160，以及 Windows 100%/125%/150%/200% 缩放组合下，正文和关键操作可读且不被裁切；
- 用户设置 150% 后，重启仍保持该比例。

建议提交：`feat: add student AI guidance and adaptive workspace UI`

### 阶段 G2：固定完整固件基线与映射契约

目标：消除当前对 `D:\RobotDog\ch32v203-robot-dog` 和开发机现成 `build/obj/compile_commands.json` 的硬编码依赖。

任务：

1. 新增固件基线清单，记录固件仓库版本、提交 SHA、板型、Flash/RAM 限制、启动文件、链接脚本和工具链版本。
2. 为开发模式提供显式固件源路径，为发布模式提供应用内置的只读固件基线；两种模式使用同一校验算法。
3. 定义学生文件到固件工程的唯一映射：
   - `Core/Src/student_control.c`；
   - `Core/Inc/student_control.h`；
   - `student-config/line-following.yaml` 生成只读 C 头文件，不直接让 YAML 进入编译器。
4. 在固件仓库增加稳定的学生控制适配层，只暴露传感器快照、巡线参数和受限控制输出，不允许学生代码直接配置 GPIO、时钟、Flash 或通信外设。
5. 构建前复制固件基线到 Studio 管理的临时目录，再叠加学生文件；禁止修改外部固件仓库和安装目录。
6. 用受控参数生成编译清单，不依赖 IDE 最近一次生成的缓存。

验收：

- 固件源目录含空格或中文时仍可构建；
- 删除外部工程的 `build/` 后仍能从干净基线构建；
- 构建不会改变外部固件仓库的任何文件；
- 修改非学生接口文件会被映射层拒绝；
- 同一基线和同一学生提交得到相同的源码输入哈希。

建议提交：`feat: add pinned firmware baseline and student overlay`

### 阶段 G3：受控代码编辑器与 AI 编译解释

目标：让能力较强的学生可以直接查看和编写应用层代码，同时复用现有安全候选、预检和 Git 恢复能力。

任务：

1. 增加“编写代码”工作台，使用 Monaco 或等价编辑器提供 C/YAML 高亮、行号、搜索、跳转、撤销和基础补全。
2. 文件导航按学生认知分组为“控制逻辑”“参数设置”“参考接口”；只允许编辑学生白名单文件，固件适配层可以只读查看，硬件与构建文件不展示或不可编辑。
3. 手动编辑进入独立 `manual draft` 候选，不直接写正式工作区；保存草稿不需要反复确认，最终“保存到项目”统一展示一次 Diff 并创建 Git 检查点。
4. AI 候选和手动草稿不能同时写同一工作区；切换前提示完成、保存或放弃当前草稿，避免静默覆盖。
5. 提供“检查代码”按钮，运行与 AI 候选相同的快速预检；错误以编辑器标记、问题列表和中文摘要同时显示。
6. 提供“请 AI 解释这个错误”，只把结构化诊断、允许文件片段、课程级别和当前提交发送给 Reasonix，不发送绝对路径、密钥或完整固件源码。
7. 提供“让 AI 尝试修复”时创建新的 AI 候选，并继续遵循一轮一次总审批；AI 不能绕过学生尚未保存的草稿状态。
8. 支持把选中代码发送给 AI 询问“这段代码在做什么”，回答默认逐段解释而不是直接重写。

验收：

- 学生能查看接口、修改 C/YAML、预检、统一确认并形成 Git 提交；
- 受保护文件无法通过编辑器、IPC 或路径技巧写入；
- 典型语法错误、类型错误、缺少声明和参数越界都有适合小学生的中文解释；
- AI 修复后仍显示完整 Diff，且能撤销回手动编辑前的提交；
- 重启后未完成草稿可以恢复或明确放弃，不产生半保存文件。

建议提交：`feat: add guarded student code editor and AI diagnostics`

### 阶段 G4：完整固件构建证明与产物管理

目标：把快速候选预检与完整固件构建明确区分，并建立可追溯产物。

任务：

1. 保留现有候选预检，用于审批前快速发现 C 语法和 YAML 范围错误。
2. 新增工作区完整构建服务，输入只能是工作区 ID 和 Git 提交，不接受 Renderer 传入任意路径或命令。
3. 构建证明记录：
   - 学生工作区提交 SHA；
   - 学生源码树哈希；
   - 固件基线 SHA；
   - 工具链和板卡配置 SHA-256；
   - ELF / HEX / BIN / MAP 哈希与大小；
   - Flash / RAM 占用；
   - 构建开始、结束和结果。
4. 构建缓存以全部输入哈希为键；工作区撤销、重新应用或固件基线变化时，旧产物自动失效。
5. 产物写入用户数据目录，先写临时目录，全部完成后原子发布；失败和取消不留下“可下载”标记。
6. 对编译错误进行结构化归类，界面只显示工作区相对路径和学生可理解的中文说明。

验收：

- 完整构建生成有效 ELF、HEX、BIN、MAP；
- 任何输入改变都会使旧 BIN 失效；
- 构建取消、进程崩溃和磁盘空间不足不会覆盖上次成功产物；
- Flash/RAM 超限时明确失败，不能进入下载步骤；
- 构建日志不泄露 API Key、用户目录或无关环境变量。

建议提交：`feat: build traceable full firmware artifacts`

### 阶段 G5：工作台教学闭环、示范与错误恢复

目标：让教师和学生不需要理解 Git、哈希或工具链术语，也能完成完整软件流程，并能随时重新观看操作示范。

任务：

1. AI 修改应用成功后，提供明确的“生成完整固件”下一步，不自动烧录。
2. “编译 / 烧录”页区分：候选预检、完整固件构建、模拟下载三个状态。
3. 产物卡片显示版本、生成时间、Flash/RAM 占用、对应存档和是否已过期。
4. 设置页补齐：Reasonix 状态、API Key 设置/清除、工具链自检、固件基线版本、用户数据目录、诊断导出。
5. 首次启动向导只要求创建训练项目和配置可选的 API Key；没有 API Key 时非 AI 功能完整可用。
6. 将内部错误码映射为中文问题、原因和下一步；终端保留技术细节，界面不再出现裸 `CANDIDATE_NOT_APPLICABLE` 一类代码。
7. 启动时对账构建任务、临时目录、工作区 HEAD 和产物证明，恢复到确定状态。
8. 学生模式采用渐进式展示：默认只显示当前任务、主要按钮和结果；Git 哈希、工具链参数、原始日志收进“查看技术细节”。
9. 新增可重复打开的操作示范中心，至少包含“让 AI 调参数”“看懂修改并应用”“自己写一行代码”“根据编译错误求助 AI”“生成完整固件”五个短任务。
10. 示范使用内置样例项目和模拟设备，采用逐步聚焦、实际点击和完成反馈；不是一次性幻灯片，也不要求连接硬件。
11. 为“候选修改、预检、编译、应用、撤销、固件”等词提供贴近机器马比赛的短解释，并保持按钮名称与完成提示一致。
12. 教师可重置示范进度，学生可跳过；跳过后不反复弹出打断正常操作。

验收：

- 新用户从创建项目到生成完整 BIN 不需要命令行；
- 重启后对话、候选、Git 历史和最后一次成功产物均可恢复；
- 重复点击、页面切换和慢速构建不会触发并发应用或重复提交；
- 所有失败状态都有可执行的下一步按钮。
- 第一次使用的学生能在示范引导下独立完成一次参数修改和一次手动代码预检；
- 学生模式不显示裸 Git、ACP、SHA-256 或 GCC 参数，教师仍可展开查看。

建议提交：`feat: complete firmware build user journey`

### 阶段 G6：自动化、故障注入与安全回归

目标：把目前的服务级测试提升到可重复的 Electron 完整流程测试。

任务：

1. 增加 Electron E2E：创建工作区、AI 对话、审批、Diff、预检、应用、完整构建、撤销。
2. 增加固定 Reasonix ACP fixture，覆盖分段输出、权限请求、超时、取消、崩溃和非法协议行。
3. 增加真实内置 WCH GCC 集成测试，但不连接目标板。
4. 增加故障注入：构建进程退出、应用被强制关闭、Git 锁、文件占用、磁盘写入失败和损坏元数据。
5. 增加中文用户名、空格路径、标准用户权限、只读安装目录和离线启动测试。
6. 对 IPC 参数、候选路径、符号链接、超大文件、二进制内容和日志脱敏持续做安全回归。
7. 增加提示词版本、统一审批、手动草稿、AI 错误解释和引导示范的回归测试。
8. 增加高 DPI 截图基线和键盘可达性检查，覆盖计划中的分辨率与缩放矩阵。

验收：

- CI 或专用 Windows 测试机可一条命令运行完整软件闭环；
- 强制终止应用后重启，不会出现半应用、假成功产物或不可撤销状态；
- 现有单元测试、类型检查、生产构建全部保持通过。

建议提交：`test: cover full desktop firmware workflow`

### 阶段 G7：Windows 离线发布包

目标：交付教师可直接安装和运行的测试版。

任务：

1. 增加 Electron 打包配置和 `dist:win` 脚本，先生成 x64 测试版安装包与便携包。
2. 将 Reasonix、沁恒 GCC/OpenOCD、板卡配置、固件基线和许可证作为固定 `extraResources` 打包。

4. 生成第三方许可证、版本清单、安装包 SHA-256 和构建信息。
5. 验证安装、覆盖升级、卸载保留用户项目、离线启动和无管理员日常运行。
6. 暂不实现自动更新；测试版先采用显式版本安装，避免课堂中途自动升级。

验收：

- 干净 Windows 11 电脑无需 Node.js、Git、MounRiver 或全局工具链即可运行；
- 能创建项目、调用已配置的 Reasonix、生成完整固件并完成模拟下载；
- 安装目录只读时所有运行时数据仍写入 `%LOCALAPPDATA%`；
- 卸载不会删除学生工作区和对话历史；
- 发布包内容、版本和哈希可追溯。

建议提交：`build: package offline Windows classroom beta`

## 5. 测试闸门

```powershell

```

从 G4 开始追加完整固件集成构建；从 G6 开始追加 Electron E2E；从 G7 开始追加安装包冒烟测试。任何阶段都不能用“界面显示成功”代替产物哈希、Git 状态和进程退出码检查。

## 6. 实施优先级与预计批次

建议按以下批次执行，避免一次改动跨越太多安全边界：

1. **批次一：G1**——补齐教学提示词、统一审批和高 DPI 基础，先减少当前使用阻力。
2. **批次二：G2**——固定固件基线和学生映射契约，为编辑器和完整构建提供真实接口。
3. **批次三：G3**——加入受控代码编辑器、手动草稿和 AI 编译错误解释。
4. **批次四：G4**——生成可追溯完整固件产物。
5. **批次五：G5**——完成学生化界面、操作示范和恢复流程。
6. **批次六：G6**——补齐 Electron E2E、显示矩阵和故障注入。
7. **批次七：G7**——制作 Windows 离线测试安装包。

G1 可以立即开始，不依赖硬件和固件接口最终决定。G2 必须先于代码编辑器和完整构建完成；没有固定固件基线和学生接口契约时，直接做编辑器、安装包或真实下载都会积累返工，甚至把当前开发机路径和临时构建缓存固化进产品。

## 7. 开始实施前需要确认的输入

以下信息不影响 G1 开始，但应在 G2 收尾前确定：

- `D:\RobotDog\ch32v203-robot-dog` 中哪一个提交作为首个固件基线；
- 该固件及沁恒库是否允许随安装包一起分发；
- 学生控制入口的最终函数签名和允许访问的数据；
- YAML 参数到固件常量的范围、默认值和单位；
- 首个测试版采用安装包、便携包，还是两者都交付。

- 学生模式允许查看哪些只读固件接口文件；
- 教师诊断模式是否需要保留逐工具审批开关。

在这些信息确定前，可以先实现清单格式、隔离构建目录、映射接口和测试夹具，但不把固件基线标记为可发布。
